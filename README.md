# -
强制加载视频到内存
================================================================================
          RAM-Stream-Browser: 基于物理内存全量预加载的多平台套壳浏览器
                            系统架构设计与实现白皮书
================================================================================
（要求全栈开发可用的APP）
一、 项目概述 (Project Overview)
--------------------------------------------------------------------------------
1.1 背景与痛点
    现代 Chromium 浏览器及流媒体平台的 Web 播放器（如 Bilibili、YouTube）受到浏览器沙盒机制
    和网页端播放脚本（Buffer Control）的严格限制，通常仅允许预加载未来 10s-60s 的视频数据。
    这导致即使用户拥有高达几百兆的带宽和 32GB/64GB 的大物理内存，在观看 4K HDR、高码率长视频
    时，依然会因为网络波动频繁出现卡顿。

1.2 核心愿景
    构建一个基于 Windows 11 环境的定制化套壳浏览器。使用户在无感保留平台原生功能（账号登录、
    好友动态、点赞三连、原生交互、弹幕机渲染）的同时，彻底接管视频流的下载与解码。利用本地
    物理内存（RAM）建立数 GB 的最高速前向与后向缓冲区，实现“秒开播放、无限缓冲、秒回倒带”。

1.3 核心技术栈选择
    * 主框架：Tauri 2.0 (基于 Rust)
      原因：相比 Electron，Tauri 产生的系统开销极小，内存占用低，且 Rust 具备无与伦比的
      系统底层交互与高并发多线程处理能力，非常适合处理大规模内存流缓冲区。
    * 内核渲染：Webview2 (基于 Chromium 核心，Windows 11 原生免安装)
    * 解析引擎：yt-dlp (以本地独立守护进程运行)
    * 解码核心：libmpv (通过 Rust 绑定原生嵌入)

二、 核心系统框架架构 (System Architecture)
--------------------------------------------------------------------------------
系统整体采用“双轨制视窗层叠架构”（Dual-Track View Overlapping Architecture）：

+------------------------------------------------------------------------------+
|                     Windows 11 OS / Tauri 主控制进程                           |

|  [顶部视图层：Webview2]                                                       |

|  - 保持全量登录状态、大会员 Cookie、点赞三连、社交、评论等交互                    |
|  - 核心魔改：通过 CSS 强行将原网页视频 <div> 挖空并设为完全透明 (Opacity: 0)  |
|  - 保留弹幕 Canvas 层覆盖在最上方                                            |

|  - 将 libmpv 渲染实例精准“贴”在 Webview2 透明视频区的正下方                    |
|  - 直接申请物理内存（RAM），绕过浏览器沙盒限制进行高并发分片下载与解码          |

三、 技术细节与关键实现 (Technical Implementation)
--------------------------------------------------------------------------------
3.1 权限与凭证无感继承 (Session & Cookie Inherit)
    * 机制：用户直接在 Webview2 视图内进行正常的平台登录。
    * 实现：本地 Agent 监听 Webview2 的网络请求请求头。一旦匹配到视频网的目标 URL（例如：
      *.bilibili.com 或 *.youtube.com），Rust 后台通过 Webview2 提供的 `CoreWebView2CookieManager` 
      API，实时导出当前域下的所有 `Cookie` 以及 `User-Agent`、`Sec-Ch-Ua` 等特征指纹。
    * 作用：保证 yt-dlp 在后台向视频平台服务器索要数据时，能够 100% 伪装成当前浏览器用户的
      真实请求，无感继承“大会员 4K”、“高帧率”、“HDR” 或 YouTube Premium 观影权限。

3.2 任意平台视频流拦截与提取 (Stream Interception)
    * 机制：通过网络请求拦截与网页 DOM 注入双重保险提取画质最高直链。
    * 方法 A (网络层拦截)：注册 Webview2 的 `WebResourceRequested` 事件，监控所有包含 
      `playurl`、`dash`、`m3u8` 或 `playlist.m3u8` 的 XHR/Fetch 网络请求，直接截获
      包含音视频流绝对路径的 JSON 响应。
    * 方法 B (注入脚本提取)：向网页注入 Preload 预加载脚本，直接在宿主上下文中读取全局变量。
      * Bilibili: 直接读取 `window.__playinfo__` 对象。
      * YouTube: 直接拦截播放器组件的初始化配置参数。
    * 拦截后，脚本触发带有音视频流 URL 数据的 Tauri IPC 通信，通知 Rust 主进程中止网页
      自带播放器的后续加载（调用网页 video.pause() 并移出 DOM），正式将控制权移交 libmpv。

3.3 本地 Agent (yt-dlp) 解密与反防盗链 (Anti-403 Protection)
    * B站等平台对直链有极其严苛的防盗链机制，直接请求会返回 403 错误。
    * 解决方案：本地 Agent 唤醒 yt-dlp 并传递完整的参数链：
      1. `--http-header-fields`: 动态注入由 3.1 节抓取的准确 `Cookie`、`Referer: https://www.bilibili.com` 
         和系统浏览器一模一样的 `User-Agent`。
      2. 针对音画分离（Dash 机制）的平台：yt-dlp 负责同时拉取高画质视频流（Video Track）和
         高保真音频流（Audio Track），以不落地（Non-disk-write）的方式直接管道（Pipe）输送
         给 libmpv。

3.4 极限物理内存接管（RAM Buffering）配置规范
    为了强行压榨 Windows 11 系统的物理内存，主进程在初始化 `libmpv` 实例时，必须传入
    以下核心底层参数（等效于高级 mpv.conf 配置）：

    ```rust
    // Rust 伪代码：初始化 libmpv 属性配置
    mpv.set_property("cache", "yes")?; // 强制开启网络流内存缓存
    mpv.set_property("demuxer-max-bytes", "4294967296")?; // 强行分配 4GB 物理内存作为前向缓冲区上限
    mpv.set_property("demuxer-max-back-bytes", "2147483648")?; // 分配 2GB 物理内存作为后向历史缓冲区
    mpv.set_property("demuxer-readahead-defs", "3600")?; // 强制允许预加载未来长达 1 小时的视频分片
    mpv.set_property("network-timeout", "60")?; // 网络超时延长至 60 秒
    ```
    * 运行表现：只要视频一开播，后台并发线程会以用户网络带宽的绝对极限（跑满千兆网卡/Wi-Fi 6），
      疯狂向 RAM 内存写入数据。通常在开播 10s 内，整部 2GB 的 4K 电影就会彻底常驻在内存中。

3.5 弹幕机同步与交互对齐 (Subtitle & Danmaku Synchronization)
    * 由于网页原生的 Canvas 弹幕层依然在最顶层的 Webview2 中渲染，必须确保“底层的画面”与
      “顶层的弹幕”在毫秒级内同步。
    * 实现方案：
      1. libmpv 触发高频定时器（如每 16.7ms，对应 60FPS），将当前高精度播放时间戳（Time-stamp）
         通过 Tauri IPC 实时发送至 Webview2 前端。
      2. 前端劫持 B站原生的播放器时间轴函数，强制将弹幕机的渲染时钟指针与 Rust 传来的
         libmpv 时间戳绑定。
      3. 网页端的暂停、快进、点赞等操作，统一通过自定义的拦截器捕获，转换为 IPC 指令控制
         底层 libmpv 的状态，从而维持完美的双向同步。

四、 方案对比评估 (Evaluation)
--------------------------------------------------------------------------------
本套壳浏览器方案（方案二）与传统的油猴脚本调用独立 MPV 窗口方案（方案一）对比：

| 评估维度          | 方案一：油猴插件 + 独立 MPV 窗口         | 方案二：本地 Agent 套壳浏览器 (本方案) |
| :---------------- | :--------------------------------------- | :------------------------------------ |
| **全量功能无感保留**| 较差。无原生态交互、看不了网页弹幕与评论。 | 完美。保留全量原网页交互与弹幕。       |
| **多平台兼容能力** | 需为不同平台寻找不同的油猴脚本，割裂严重。 | 极强。通过统一的拦截逻辑适配多平台。   |
| **性能及内存换流畅**| 极佳。MPV 纯本地解码，开销小。           | 极佳。利用 libmpv 完美掌控系统 RAM。   |
| **自动化与自我优化**| 需要手动处理环境、路径和依赖升级。       | 本地 Agent 可以全自动静默更新后台组件。|
| **开发与维护复杂度**| 极低，即装即用。                         | 较高，需要维护视窗坐标对齐与同步逻辑。  |

五、 多平台兼容性扩展设计 (Multi-Platform Extension)
--------------------------------------------------------------------------------
为使本框架兼容 Bilibili、YouTube、Netflix、巴哈姆特动画疯等主流平台，本地 Agent 需要
构建动态“平台适配器契约”（Platform Adapter Contract）：

1. 路由拦截字典：
   * `bilibili.com/video/` -> 激活 B站流抓取器 -> 提取 `__playinfo__` -> 绑定原生 Canvas 弹幕
   * `youtube.com/watch` -> 激活 YT 流抓取器 -> 解密 YouTube VSS 签名 -> 屏蔽自带播放器
2. 针对特殊加密平台（如 Netflix / Prime Video）：
   * 由于这类平台部署了硬核 DRM（数字版权管理，如 Widevine L1/L3），libmpv 无法直接通过
     直链抓取解密。
   * 降级处理机制：Agent 一旦识别到 DRM 限制平台，自动关闭 libmpv 托管层，将播放权交还给
     Webview2 浏览器内核原生播放（由于系统为 Windows 11，Webview2 天然支持 PlayReady 和
     Widevine DRM），以确保基本可用性。

六、 部署与系统配置优化指南 (Windows 11 Deployment)
--------------------------------------------------------------------------------
为了让 Windows 11 能够完美发挥该套壳浏览器的性能，本地 Agent 需对系统底层进行如下调优：

1. GPU 硬件加速白名单：
   Tauri 主进程在创建窗口时，必须向 Chromium 内核附加命令行开关，确保 4K HDR 视频通过
   核显/独显进行硬解码（HW Decoding），避免占用过多 CPU：
   `app.commandLine.appendSwitch("ignore-gpu-blacklist");`
   `app.commandLine.appendSwitch("enable-gpu-rasterization");`
   `app.commandLine.appendSwitch("enable-zero-copy");`

2. Windows 内存分页优化：
   由于程序会强行申请数 GB 的大内存块，为防止被 Windows 11 的系统虚拟内存（Pagefile）
   强制写入速度极慢的硬盘（导致二次卡顿），本地 Agent 需要向 Windows 申请高权限的
   `VirtualLock`（内存锁定）API，将分配的缓存区块锁定在真正的**物理内存条**中，禁止页交换。
