================================================================================
          RAM-Stream-Browser: 务实版本地大缓存播放器 (MVP阶段)
                         系统架构与迭代实施方案 (v2.0)
================================================================================

一、 项目愿景与务实目标 (Pragmatic Goals)
--------------------------------------------------------------------------------
1.1 痛点重定义
    浏览器的流媒体播放受限于沙盒缓存机制（通常仅缓存数十秒），导致高码率视频易卡顿。
    目标不再是“瞬间下载全片”（受限于平台限速和真实物理带宽），而是建立一个“深度且智能的
    物理内存缓冲区（例如：提前缓存后30分钟，保留前10分钟）”，彻底屏蔽常规网络波动。

1.2 MVP 阶段范围 (Scope)
    * 核心平台：仅针对 Bilibili (B站) 进行深度适配。
    * 扩展预留：底层接口保留 YouTube 等平台的热更新挂载点，但不作为一期目标。
    * 绝对红线：不碰 Netflix、Prime 等强 DRM 平台，避免陷入无底洞的解密对抗。

1.3 核心技术栈 (保持不变)
    * 宿主框架：Tauri 2.0 (Rust) - 负责极低开销的系统进程管理与 IPC。
    * 浏览器层：Webview2 (Chromium) - 负责加载页面、鉴权、弹幕层与 UI。
    * 底层播放：libmpv (C/Rust Bindings) - 负责真正的视频解码与巨量内存吃进。

二、 柔性架构设计 (Flexible Architecture)
--------------------------------------------------------------------------------
鉴于 B站前端 DOM 结构（如播放器容器 CSS 类名）会高频变更，项目必须采用“柔性挂载”与“优雅降级”设计。

+------------------------------------------------------------------------------+
|                        Tauri 主控制进程 (Rust)                               |
|                                                                              |
|  [顶层：Webview2 网页层]                                                     |
|  - 正常加载 B站。所有的交互、评论、动态照常运作。                            |
|  - 动态注入脚本：不写死 CSS 类名，而是通过遍历 DOM 树寻找 <video> 标签。     |
|  - 优雅降级机制：如果脚本找不到特征 DOM，或者提取直链失败，插件静默，        |
|    保留 B站原生 HTML5 播放器继续播放（保证软件永远“可用”）。                |
|                                                                              |
|  [底层：libmpv 渲染层]                                                       |
|  - 一旦成功接管，将原生 <video> 设为透明（不删除，避免触发前端报错）。       |
|  - libmpv 窗口精准对齐 Webview2 中透明区域的绝对坐标。                       |
|                                                                              |
+------------------------------------------------------------------------------+

三、 核心功能点与容错实现 (Implementation & Fallbacks)
--------------------------------------------------------------------------------
3.1 直链提取与鉴权 (Bilibili First)
    * 策略：B站的 `window.__playinfo__` 目前相对稳定。
    * 实施：注入 Content Script，一旦侦测到 `__playinfo__`，立刻提取高画质 Dash 流 
      (m4s 格式的 Video 和 Audio 链接)。
    * 身份透传：利用 Webview2 API 抓取当前 Bilibili 域下的所有 Cookie，一并传给后端的 
      MPV 引擎，完美继承大会员权限。

3.2 务实的内存接管配置 (Smart RAM Buffering)
    放弃“强制塞满 4GB”，改为“智能区间缓冲”。在 libmpv 初始化时写入：
    ```ini
    cache=yes
    # 前向缓冲：约允许缓存 30 分钟的 4K 视频数据 (约 1.5GB)
    demuxer-max-bytes=1572864000
    # 后向缓冲：保留已播放的 10 分钟数据用于秒回倒带 (约 500MB)
    demuxer-max-back-bytes=524288000
    ```
    * 注：受限于 B站服务端的单连接限速（Throttling），MPV 会以稳健的速度逐步吃满这 1.5GB 
      内存，而不是瞬间拉满带宽。这依然能提供远超网页端的流畅体验。

3.3 弹幕同步难题与折中方案 (Danmaku Sync Strategy)
    * 难点：由于拖拽、倍速等操作，底层 MPV 的时间戳与顶层 Web 弹幕引擎容易产生毫秒级漂移。
    * 一期务实方案：
      1. Rust 以 60Hz 频率通过 IPC 向 Web 侧发送 MPV 的精确 `time-pos`。
      2. 前端 JS 劫持 B站原生播放器的 `currentTime` 属性，强制将其指向 Rust 传来的时间。
      3. 牺牲边缘体验：在进行大幅度拖拽进度条时，先暂停 MPV，等弹幕层完成清空和重载后，
         再恢复 MPV 播放。容忍百毫秒级的视觉延迟，换取整体系统的稳定性。

四、 应对平台演进的扩展性设计 (Maintainability)
--------------------------------------------------------------------------------
全栈开发最重维护。为了应对 B站和 YouTube 频繁更改签名和规则，系统需要剥离“规则解析”模块。

4.1 规则热更新机制 (Hot-Update Config)
    * 痛点：如果把寻找 DOM 节点、提取链接的逻辑写死在 Tauri 的 Rust 代码里，每次 B站改版 
      都需要重新编译和发布新版本。
    * 设计：
      提取规则、DOM 选取器（Selector）全部以外置的 `.json` 或 `.js` 脚本形式存在。
      程序启动时，先从你的云端仓库拉取最新的 `rules.json`。B站前端一改，你只需要更新云端 
      JSON，所有客户端下一次启动即可自适应修复。

4.2 为 YouTube 等平台预留的插件化接口 (yt-dlp Port)
    * 尽管一期不做 YouTube，但底层必须留好对接 `yt-dlp` 守护进程的通道。
    * YouTube 的 `player.js` 签名破解极难，不要自己写代码，唯一的出路是依赖 `yt-dlp` 开源 
      社区的更新。
    * 架构预留：当 `rules.json` 判定当前域名为 YouTube 时，URL 不再由自己解析，而是直接 
      通过标准输入输出 (stdin/stdout) 丢给本地的 `yt-dlp.exe`，让它返回真实的流地址给 MPV。

五、 开发实施路径 (Action Plan for Agent)
--------------------------------------------------------------------------------
不要试图让 Agent 一次性写出整个软件。请按照以下三个 Sprint (冲刺) 推进：

[Sprint 1: 最小核心跑通] 
- 任务：用 Tauri + Webview2 打开 B站，能登录，能正常播放原生网页视频。
- 验收：写一个按钮，点击后能利用 Rust 在终端打印出当前网页的 `Cookie` 和 `__playinfo__`。

[Sprint 2: 内存剥离测试]
- 任务：拿到 `__playinfo__` 中的直链后，前端屏蔽原生播放器。Rust 启动外置的 libmpv 窗口 
  (先不苛求完美覆盖重合)，传入链接和 Cookie，验证 MPV 是否能无 403 报错地播放并吃进内存。

[Sprint 3: 视窗缝合与同步] (最难)
- 任务：计算 Webview2 中视频区域的相对坐标，利用 Win32 API 动态移动 libmpv 窗口使其贴合。
- 建立 IPC 通信通道，打通 MPV 进度条与 B站弹幕的时间戳同步。
